# Phase 3 Test Report: Semantic Search & Hybrid Retrieval

## 1. 개요
- **목표**: 키워드 검색(FTS5)과 의미론적 검색(Vector Search)을 결합하여 검색 품질을 획기적으로 개선한다.
- **주요 성과**:
    - `sqlite-vec` 기반 벡터 DB 연동 및 `id_mapping`을 통한 문자열 ID 호환성 확보.
    - `digest` 수행 시 자동으로 임베딩을 생성하는 **Auto-embedding** 파이프라인 구축.
    - **RRF(Reciprocal Rank Fusion)** 알고리즘을 적용한 하이브리드 검색 구현.

## 2. 테스트 환경
- **저장소**: `pallets/click` (GitHub 공개 저장소)
- **추론 모델**:
    - 요약: `ollama/qwen2.5:3b`
    - 임베딩: `ollama/nomic-embed-text`
- **DB 상태**: 약 3,000개 이상의 이슈 및 댓글 수집 완료.

## 3. 테스트 시나리오 및 재현 명령어

### 3.1. 환경 준비 및 모델 설정
```bash
# 임베딩 모델 등록 (첫 등록 시 자동으로 'default' 에일리어스가 부여됨)
direnv exec . uv run octopith embedding-model use nomic-embed-text

# 모델 목록 확인
direnv exec . uv run octopith embedding-model list
```

### 3.2. 통합 가공 및 자동 임베딩 실행
```bash
# 요약과 임베딩을 한 번에 실행 (기본적으로 embed가 포함됨)
direnv exec . uv run octopith digest --duration 1m
```

### 3.3. 하이브리드 검색 검증
키워드 단독 검색과 하이브리드 검색의 결과를 비교합니다.

```bash
# 1. 키워드 검색 전용 (FTS5)
direnv exec . uv run octopith search "rate limit" --fast

# 2. 하이브리드 검색 (FTS5 + Vector + RRF)
direnv exec . uv run octopith search "rate limit" --limit 5
```

## 4. 테스트 결과

### 4.1. 검색 품질 비교
- **키워드 검색**: "rate limit" 단어가 정확히 포함된 제목/본문 위주로 노출.
- **하이브리드 검색**: "API exhaustion", "throttling" 등 직접적인 단어가 없더라도 의미적으로 연관된 이슈들이 상위에 랭크됨 (`SEMANTIC_THREAD` 매치 타입 확인).

### 4.2. RRF 랭킹 시스템
- 동일한 이슈가 키워드와 시맨틱 모두에서 발견될 경우, `HYBRID` 매치 타입으로 표시되며 높은 점수(Score)를 획득함을 확인.

## 5. 결론
Phase 3의 모든 요구사항이 충족되었으며, 특히 `digest`와 임베딩의 자동 통합으로 인해 사용자 편의성이 극대화되었습니다. 하이브리드 검색은 단순 키워드 검색보다 훨씬 풍부한 결과를 제공합니다.
