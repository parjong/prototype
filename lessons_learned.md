# Octopith 의미 기반 검색 엔진 (PoC) 교훈 및 이슈 회고

본 문서는 Octopith의 시맨틱 검색 파이프라인(요약 및 임베딩)을 구축하는 과정에서 직면했던 주요 문제들과 초기 가정들, 그리고 이를 어떻게 해결했는지에 대한 상세한 기록입니다. 향후 이 PoC를 바탕으로 운영 환경을 설계할 때 중요한 지침이 됩니다.

---

## 1. 초기 가정 및 한계
- **가정 1**: "로컬 소형 임베딩 모델(`nomic-embed-text`)이라도 한국어로 된 문서를 그대로 임베딩하면 영어 검색어("self-improving" 등)와 의미적으로 매칭될 것이다."
  - **실제**: 다국어(Multilingual) 전용 거대 모델이 아닌 이상, 모델은 한/영 교차 검색 능력이 현저히 떨어졌습니다. 동일한 의미라도 언어가 다르면 임베딩 공간 상에서 거리가 멀어 매칭에 실패했습니다.
- **가정 2**: "요약본 벡터 검색 결과와 본문 벡터 검색 결과를 리스트로 단순히 이어붙이고(Union) RRF를 적용하면, 알아서 적절한 순위가 매겨질 것이다."
  - **실제**: 쿼리 결과를 합칠 때 거리(Distance) 기준의 전역 정렬(Global Sort)을 누락할 경우, RRF 수식의 랭킹 가중치가 꼬이게 됩니다.

---

## 2. 주요 발생 이슈 및 해결 과정

### 2.1 다국어 검색 한계 우회 (Language Barrier Bypass)
- **문제 상황**: "Self-improving"이라는 영문 검색어로 검색했을 때, 한국어로 작성된 "자율 루프(Self-analyzing)" 관련 이슈가 전혀 검색되지 않았습니다.
- **해결 방식**: 
  - 로컬 임베딩 모델의 언어적 한계를 극복하기 위해, **문서 요약 단계에서 LLM(`qwen2.5:3b`)에게 "모든 문서를 무조건 영어로 요약하라"는 강력한 프롬프트 지시어(Mandatory English Output)를 추가**했습니다.
  - 결과적으로 DB에는 한국어 이슈에 대한 고품질의 영어 요약본이 저장되었고, 이 영어 요약본들이 임베딩되면서 영문 쿼리와의 의미 기반(Semantic) 교차 검색이 완벽하게 동작하게 되었습니다.

### 2.2 모델 별칭 중복으로 인한 차원 불일치 (Dimension Mismatch Error)
- **문제 상황**: 
  - 벡터 검색 도중 `Dimension mismatch for query vector... Expected 768 dimensions but received 2048` 에러가 발생했습니다.
  - 임베딩 생성 시 768차원 모델(`nomic-embed-text`)이 사용되어야 하는데, 요약 모델(`qwen2.5:3b`)이 임베딩을 수행해버려 2048차원의 벡터가 DB에 저장/조회되었습니다.
- **원인 분석**: 
  - DB의 `ModelAlias` 테이블에서 `"default"`라는 별칭 하나가 요약 모델과 임베딩 모델 양쪽에서 모호하게 사용되고 있었습니다. `get_model_variant_by_alias("default")`를 호출할 때 모델의 타입(Type)을 구분하지 않고 무조건 첫 번째 매칭(주로 요약 모델)을 반환했습니다.
- **해결 방식**: 
  - CLI 명령어(`digest`, `search`) 레벨에서 임베딩 모델의 기본 별칭을 `"nomic-embed-text"`로 명시하도록 강제하여 모델 꼬임 현상을 방지했습니다. 
  - *(향후 과제)*: DB 수준에서 Alias 조회 시 모델 `Type`을 반드시 포함하여 검색하도록 스키마 및 쿼리 설계를 보강해야 합니다.

### 2.3 벡터 검색 결과 병합 시 정렬 누락 버그 (Sorting Bug Before RRF)
- **문제 상황**: 언어 문제와 차원 문제를 모두 해결했음에도 불구하고, 핵심 요약본(`companion#111`)이 검색 결과 상위 10개 안에 노출되지 않았습니다.
- **원인 분석**:
  - `store.py`의 `search_vector()` 함수에서 SQLite(`vec_threads`)의 검색 결과 10개와 (`vec_summaries`)의 검색 결과 10개를 빈 리스트에 단순히 `.append()`로 이어 붙였습니다.
  - 전체 리스트를 거리(Distance) 순으로 다시 정렬하지 않은 채 반환했기 때문에, 뒤에 덧붙여진 요약본들은 거리가 매우 가깝더라도 리스트의 11~20번째 인덱스에 위치했습니다.
  - 이후 RRF 점수 산정 시 인덱스(Rank) 번호가 그대로 패널티로 작용하여 최종 점수가 폭락했습니다.
- **해결 방식**:
  - 두 테이블의 검색 결과를 리스트로 합친 직후, `results.sort(key=lambda x: x["distance"])` 코드를 추가하여 **전역(Global) 기준의 거리 오름차순 정렬**을 보장했습니다. 이를 통해 요약본이 상위 1위로 정상 노출되었습니다.

### 2.4 하드코딩된 유사도 임계값 (Hardcoded Distance Threshold)
- **문제 상황**: 초기 설계에서 SQLite KNN 쿼리에 `distance < 0.3`이라는 엄격한 필터 조건을 하드코딩해 두었습니다.
- **문제점**:
  - `nomic-embed-text` 모델 특성상 의미적으로 유사한 문장("self-analyzing"과 "self-improving")이라도 거리가 `0.375` 등으로 측정될 수 있습니다. 
  - 엄격한 필터로 인해 유효한 의미 매칭 결과가 원천 차단되었습니다.
- **해결 방식**:
  - 임계값을 `0.6`으로 완화했습니다. 
  - *(향후 과제)*: 사용되는 임베딩 모델의 특성에 맞게 Threshold를 동적으로 설정하거나, Config 테이블을 통해 관리할 수 있는 구조가 필요합니다.

## 3. 향후 아키텍처 설계를 위한 제언
1. **번역-임베딩 파이프라인 공식화**: 다국어 검색을 위해 본문과 요약본의 **언어 통일(Normalization to English)** 전략은 매우 유효합니다. 향후 정식 구조에서는 `LanguageNormalizer` 같은 독립된 전처리 파이프라인 모듈로 분리하는 것이 좋습니다.
2. **별칭(Alias)과 설정 관리의 엄격성**: 모델 Alias는 `(Type, Alias)`의 복합 키(Composite Key) 형태로 관리하거나, CLI 레벨이 아닌 Store 내부에서 엄격하게 Type Validation을 수행해야 합니다.
3. **RRF 병합 컴포넌트의 단일화**: 현재 Store 안에 하드코딩된 RRF(Reciprocal Rank Fusion) 로직을 별도의 `Ranker` 또는 `Scorer` 클래스로 분리하여, 향후 BM25(FTS) 가중치나 Vector 거리 가중치를 외부에서 쉽게 주입/테스트할 수 있도록 개선해야 합니다.
